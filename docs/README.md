## ドキュメント活用フロー

### 新規開発者向けの推奨読書順序

1. **[要件定義書](./requirements.md)** - プロジェクト全体を理解
2. **[アーキテクチャ設計書](./architecture.md)** - システム構成を把握
3. **[実装ガイド](./implementation-guide.md)** - 開発環境をセットアップ
4. **[技術仕様書](./technical-spec.md)** - 使用技術の詳細を確認
5. **[UI/UX設計書](./ui-ux-design.md)** - デザインガイドラインを理解

### 機能追加時の参照フロー

1. **[要件定義書](./requirements.md)** - 機能要件を確認
2. **[アーキテクチャ設計書](./architecture.md)** - コンポーネント設計を参照
3. **[UI/UX設計書](./ui-ux-design.md)** - UI仕様を確認
4. **[実装ガイド](./implementation-guide.md)** - 実装手順に従う
5. **[変更履歴](./changelog.md)** - リリース時に更新

### トラブルシューティング時の参照フロー

1. **[実装ガイド](./implementation-guide.md)** - よくある問題を確認
2. **[技術仕様書](./technical-spec.md)** - 技術詳細を再確認
3. **[API仕様書](./api-spec.md)** - API関連エラーの場合

## ドキュメント更新ルール

### 更新タイミング

| ドキュメント | 更新タイミング | 担当者 |
|------------|--------------|-------|
| 要件定義書 | 機能要件変更時、新機能追加時 | PM、開発リーダー |
| 技術仕様書 | 技術スタック変更時、ライブラリアップデート時 | 開発リーダー |
| アーキテクチャ設計書 | コンポーネント追加時、大規模リファクタリング時 | 開発者 |
| UI/UX設計書 | デザイン変更時、新UIコンポーネント追加時 | デザイナー、開発者 |
| 実装ガイド | 開発プロセス変更時、新ツール導入時 | 開発者 |
| API仕様書 | API連携追加時、仕様変更時 | 開発者 |
| 変更履歴 | 毎リリース時 | リリース担当者 |

### 更新手順

1. ドキュメントを編集
2. 最終更新日を更新
3. 変更履歴セクションに記載（該当する場合）
4. プルリクエストで変更をレビュー
5. マージ後、チームに共有

## ドキュメントのベストプラクティス

### 記述ルール

- **明確な見出し**: セクションごとに明確な見出しを使用
- **具体例**: コード例やスクリーンショットを含める
- **簡潔性**: 必要な情報のみを記載し、冗長を避ける
- **一貫性**: 用語、フォーマットを統一
- **リンク**: 関連ドキュメント間でリンクを活用

```markdown

**太字**: 重要なキーワード
*斜体*: 強調
```
